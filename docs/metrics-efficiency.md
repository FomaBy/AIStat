# Метрики эффективности агентов

Документ фиксирует три метрики эффективности AIStat и, отдельно, формулу и
обоснование **взвешенной эффективности** (FAN-1227). Все три считаются в
`aistat/aggregates.py` (`efficiency_breakdown`, `summary`) и отдаются в
`GET /api/summary` (карточки) и `GET /api/model-efficiency` (разрез по моделям).
Графики токеновой эффективности по агентам, моделям и времени отдаёт
`GET /api/efficiency-breakdown`.

## Общий принцип отбора задач

Все метрики считаются только по задачам, которые:

- не импортированы из Jira (`is_jira = 0`);
- имеют story points > 0;
- имеют загруженную статистику токенов (`issue_usage`).

Задачи без story points **исключаются** из метрик целиком (не считаются как SP=0).
Фильтры проекта, агента, модели и периода действуют и на эффективность. Для
агента, модели или временного окна накопленные токены и story points задачи
получают одинаковую долю подтверждённых датированных запусков; результат такой
атрибуции помечается как оценочный (`≈` / `estimated`): `/api/summary` отдаёт
флаги `sp_estimated` и `efficiency_estimated` для карточек SP и токеновой
эффективности, `/api/efficiency` — флаг `estimated` на строку задачи. Без
фильтров task-level значения точные и не помечаются.

Стоимость и «взвешенная» метрика дополнительно требуют, чтобы у задачи были
запуски (`runs`): без запусков нельзя привязать точные токены задачи к модели и
тарифицировать их. Токены, попавшие на модель без официального тарифа, выводятся
в `unpriced_tokens` и помечаются флагом, но никогда не оцениваются в $0.

**Единый отфильтрованный набор запусков (FAN-1244).** При фильтре по агенту,
модели или времени все метрики строятся из одного и того же набора пересечений
запусков: каждый запуск сверяется с фильтрами агента/модели и пересекается с
временным окном ровно один раз. Из этого набора берутся сразу доля токенов/SP
задачи, состав моделей, веса моделей, стоимость и активные часы:

- модели запусков, исключённых фильтром (например, модель другого агента при
  `agent=`-фильтре), в разрез и в стоимость не попадают;
- активные часы равны фактическому времени пересечения запусков с окном —
  общая доля фильтра к уже отфильтрованным часам повторно не применяется;
- знаменатели `cost_per_sp` и `weighted_efficiency` — SP отфильтрованного
  подмножества, а не lifetime-статистика задачи × доля;
- отобранный запуск без model metadata (агент без модели или запуск без
  агента) сохраняет собственную долю в наборе как строку `model = null`
  (FAN-1247): его часы и доля токенов/SP не перераспределяются на известные
  модели, а тарифицируются как неизвестная модель — попадают в
  `unpriced_tokens` / `has_unpriced`, и задача с такой долей исключается из
  `weighted_efficiency` как не полностью тарифицированная. Полностью
  model-less выборка отдаёт `cost_per_sp = weighted_efficiency = null` и всю
  долю в `unpriced_tokens`, никогда не $0 и не чужую модель.

Без фильтров и при фильтре только по проекту используется полная статистика
запусков задачи — эти значения не меняются.

## 1. Эффективность токенов (`tokens_per_sp`)

Бывшая «Эффективность». Переименована в «Эффективность токенов», чтобы отличать
её от новых стоимостных метрик.

```
tokens_per_sp = Σ tokens_i / Σ sp_i
```

Точные токены задач ÷ сумма story points. Меньше — лучше. Считается по всем
отобранным задачам (запуски не обязательны, токены известны из `issue_usage`).

### Графики по агентам, моделям и времени

`GET /api/efficiency-breakdown` строит графики именно этой метрики. Для каждой
задачи с валидными `started_at` и `completed_at` её точные накопленные токены и
SP делятся между запусками пропорционально их длительности. Важное правило для
среза модели: `SP модели = SP задачи × длительность запусков модели /
длительность всех валидных запусков задачи`; к токенам применяется та же доля.
Выбор одного агента, модели или короткого окна не перенормализует знаменатель,
поэтому час работы не выглядит как вся задача.

График времени использует UTC-часы только для явно заданного интервала не более
48 часов; для остальных диапазонов — UTC-дни. Задачи без валидных датированных
запусков не получают выдуманную атрибуцию и в эти три графика не входят.

Пустой отфильтрованный результат дашборд показывает явным сообщением «Нет
данных…» поверх графика, а не пустым полотном. Интервалы без атрибуции остаются
разрывами: на графике времени `tokens_per_sp = null` не соединяется линией и не
превращается в 0. Значения каждого графика дублируются раскрываемой таблицей
(«Данные таблицей»), доступной с клавиатуры и через screen reader; сами canvas
имеют доступное имя (`role="img"` + `aria-label`).

## 2. Эффективность стоимости (`cost_per_sp`)

Сколько денег (USD) стоит один story point.

```
cost_per_sp = Σ cost_i / Σ sp_i          (только по тарифицированным долям)
```

`cost_i` — оценочная стоимость точных токенов задачи: токены делятся между
моделями, выполнявшими задачу (по длительности запусков), и тарифицируются по
официальной ставке каждой модели (`aistat/pricing.py`).

**Числитель и знаменатель берутся из одних и тех же тарифицированных долей
(QA FAN-1188).** В знаменатель попадает SP только той доли задачи, чью стоимость
удалось привязать к тарифу. Доля на модели без тарифа (в т.ч. `model = null`) не
добавляет свою SP в знаменатель против нулевой стоимости, иначе частично или
полностью нетарифицированная задача выглядела бы дешевле, чем была. Практически:

- полностью нетарифицированная задача в `cost_per_sp` не участвует вовсе (ни
  числителем, ни знаменателем) — её SP не разбавляет метрику;
- у частично тарифицированной задачи в метрику входит только тарифицированная
  доля стоимости и ровно её же доля SP; нетарифицированная доля идёт в
  `unpriced_tokens` / `has_unpriced` и в `cost_per_sp` не учитывается;
- модель с нулевым attribution weight (например, запуск нулевой длительности
  рядом с реальным) не участвует ни в числителе, ни в знаменателе и не помечает
  задачу как нетарифицированную;
- задача с SP и usage, но без запусков, не атрибутируется ни к одной модели и в
  стоимостную метрику не входит (в токеновую — входит).

Полностью нетарифицированная выборка отдаёт `cost_per_sp = null`, никогда `$0`.

**Разрез по моделям.** `GET /api/model-efficiency` возвращает `cost_per_sp` для
доли каждой модели: SP и токены задачи делятся между моделями по длительности их
запусков, стоимость — по тарифу модели. Под фильтром агента/модели/времени
длительности — это пересечения только отобранных запусков с окном, поэтому
модель, все запуски которой исключены фильтром, в разрезе не появляется. Строки
отсортированы по стоимости на 1 SP (дешевле сверху), поэтому сразу видно, какая
модель делает story point дешевле.

## 3. Взвешенная эффективность (`weighted_efficiency`)

Коэффициент, показывающий, насколько экономно был сделан story point с учётом
времени: **сколько денег в час тратится на один story point** (USD / час / SP).
Меньше — лучше.

### Формула

Для одной задачи `i`:

```
we_i = cost_i / hours_i / sp_i
```

где `hours_i` — активное время запусков задачи (сумма
`completed_at − started_at` по запускам, в часах; под фильтром
агента/модели/времени — сумма фактических пересечений отобранных запусков с
окном). Агрегируется как среднее, взвешенное по story points:

```
weighted_efficiency = Σ (sp_i · we_i) / Σ sp_i = Σ (cost_i / hours_i) / Σ sp_i
```

В расчёт входят только полностью тарифицированные задачи с `hours_i > 0` и
`sp_i > 0` (чтобы `cost_i` был полным). Разрез по моделям использует ту же
формулу для доли модели: `cost` и `hours` — доля модели по данной задаче, `sp` —
доля SP модели.

### Обоснование выбора

- **Единицы совпадают с формулировкой задачи** — «деньги в час на 1 SP»:
  `(USD / час) / SP`. И «в час», и «на 1 SP» стоят в знаменателе.
- **Масштаб-инвариантность.** Прямое `Σcost / (Σhours · Σsp)` было бы обманчивым:
  и `Σhours`, и `Σsp` растут вместе с объёмом работы, поэтому у крупного проекта
  или модели, сделавшей больше SP, коэффициент искусственно занижался бы — их
  нельзя было бы честно сравнивать. SP-взвешенное среднее по задачам
  (`Σ(cost_i/hours_i)/Σsp_i`) при однородных данных равно `we_i` одной задачи и
  не зависит от числа задач, поэтому модели, проекты и периоды сравниваются
  корректно (именно это и нужно для «какая модель делает SP дешевле»).
- **Направление «меньше — лучше»** согласовано с остальными метриками: ниже
  коэффициент — меньше денег в час тратится на story point. Метрика поощряет
  ровный, недорогой темп расхода; всплеск затрат за короткое время повышает
  коэффициент.

### Ограничения

- Стоимость — оценка (атрибуция токенов по моделям + официальные тарифы), поэтому
  в UI метрика всегда помечается `≈`.
- Задачи без запусков или без тарифа модели не входят в стоимостные метрики;
  наличие неоценённых токенов помечается флагом (`has_unpriced`, `*`).
- `hours_i` считается по времени запусков; запуск, пересекающий полночь,
  учитывается целиком (то же приближение, что и в атрибуции токенов).

## Количество и время работы агентов (FAN-1228)

Отдельно от трёх метрик выше `summary` считает **участие агентов** и **суммарное
время их работы**; они не ограничены SP/usage-eligible задачами, поэтому не
совпадают с `efficiency_hours`.

- `agent_work_seconds` = сумма длительностей *подходящих* запусков, обрезанных по
  выбранному окну (`_agent_work_seconds` в `aistat/aggregates.py`, общий helper
  `_overlap_seconds`). `agent_count` — число различных агентов с подходящей
  работой. `/api/agents` отдаёт ту же величину на агента в поле `work_seconds`;
  при одинаковых фильтрах сумма `work_seconds` по агентам равна
  `agent_work_seconds`, а число агентов с работой — `agent_count`.
- **Подходящий запуск** — строка `runs` с непустым `agent_id`, валидными UTC
  `started_at`/`completed_at` и `completed_at > started_at`. Completed, failed и
  cancelled учитываются; pending/dispatched/running и
  отсутствующие/некорректные/нулевые/отрицательные интервалы — нет.
- **Окно** полузакрытое `[from, to)`: запуск даёт только своё пересечение с
  окном, граница даёт ноль, date-only `to` остаётся inclusive-day. Retry — это
  отдельный `run.id`; одновременные запуски складываются как время агента, а
  дублирующийся `run.id` (из `agent tasks` и `issue runs`) считается один раз.
- **Единицы** — секунды (запуски имеют секундную точность), это каноническое
  значение API; дашборд показывает читаемую длительность.
- **Отличие от `efficiency_hours`.** `efficiency_hours` — активные часы только по
  задачам с SP и `issue_usage` (см. «Общий принцип отбора задач»); агентское
  время считает любой подходящий запуск и агрегируется по агентам, а не по
  задачам. Значения не свежее последнего цикла поллера.
