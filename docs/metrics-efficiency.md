# Метрики эффективности агентов

Документ фиксирует три метрики эффективности AIStat и, отдельно, формулу и
обоснование **взвешенной эффективности** (FAN-1227). Все три считаются в
`aistat/aggregates.py` (`efficiency_breakdown`, `summary`) и отдаются в
`GET /api/summary` (карточки) и `GET /api/model-efficiency` (разрез по моделям).

## Общий принцип отбора задач

Все метрики считаются только по задачам, которые:

- не импортированы из Jira (`is_jira = 0`);
- имеют story points > 0;
- имеют загруженную статистику токенов (`issue_usage`).

Задачи без story points **исключаются** из метрик целиком (не считаются как SP=0).
Фильтр периода на метрики эффективности не действует — story points и
эффективность это атрибут задач проекта, а не дневной ряд; действует только
фильтр проекта.

Стоимость и «взвешенная» метрика дополнительно требуют, чтобы у задачи были
запуски (`runs`): без запусков нельзя привязать точные токены задачи к модели и
тарифицировать их. Токены, попавшие на модель без официального тарифа, выводятся
в `unpriced_tokens` и помечаются флагом, но никогда не оцениваются в $0.

## 1. Эффективность токенов (`tokens_per_sp`)

Бывшая «Эффективность». Переименована в «Эффективность токенов», чтобы отличать
её от новых стоимостных метрик.

```
tokens_per_sp = Σ tokens_i / Σ sp_i
```

Точные токены задач ÷ сумма story points. Меньше — лучше. Считается по всем
отобранным задачам (запуски не обязательны, токены известны из `issue_usage`).

## 2. Эффективность стоимости (`cost_per_sp`)

Сколько денег (USD) стоит один story point.

```
cost_per_sp = Σ cost_i / Σ sp_i          (по задачам с привязкой к модели)
```

`cost_i` — оценочная стоимость точных токенов задачи: токены делятся между
моделями, выполнявшими задачу (по длительности запусков), и тарифицируются по
официальной ставке каждой модели (`aistat/pricing.py`). Знаменатель — сумма SP
задач, у которых стоимость удалось привязать.

**Разрез по моделям.** `GET /api/model-efficiency` возвращает `cost_per_sp` для
доли каждой модели: SP и токены задачи делятся между моделями по длительности их
запусков, стоимость — по тарифу модели. Строки отсортированы по стоимости на 1 SP
(дешевле сверху), поэтому сразу видно, какая модель делает story point дешевле.

## 3. Взвешенная эффективность (`weighted_efficiency`)

Коэффициент, показывающий, насколько экономно был сделан story point с учётом
времени: **сколько денег в час тратится на один story point** (USD / час / SP).
Меньше — лучше.

### Формула

Для одной задачи `i`:

```
we_i = cost_i / hours_i / sp_i
```

где `hours_i` — активное время запусков задачи (сумма
`completed_at − started_at` по запускам, в часах). Агрегируется как среднее,
взвешенное по story points:

```
weighted_efficiency = Σ (sp_i · we_i) / Σ sp_i = Σ (cost_i / hours_i) / Σ sp_i
```

В расчёт входят только полностью тарифицированные задачи с `hours_i > 0` и
`sp_i > 0` (чтобы `cost_i` был полным). Разрез по моделям использует ту же
формулу для доли модели: `cost` и `hours` — доля модели по данной задаче, `sp` —
доля SP модели.

### Обоснование выбора

- **Единицы совпадают с формулировкой задачи** — «деньги в час на 1 SP»:
  `(USD / час) / SP`. И «в час», и «на 1 SP» стоят в знаменателе.
- **Масштаб-инвариантность.** Прямое `Σcost / (Σhours · Σsp)` было бы обманчивым:
  и `Σhours`, и `Σsp` растут вместе с объёмом работы, поэтому у крупного проекта
  или модели, сделавшей больше SP, коэффициент искусственно занижался бы — их
  нельзя было бы честно сравнивать. SP-взвешенное среднее по задачам
  (`Σ(cost_i/hours_i)/Σsp_i`) при однородных данных равно `we_i` одной задачи и
  не зависит от числа задач, поэтому модели, проекты и периоды сравниваются
  корректно (именно это и нужно для «какая модель делает SP дешевле»).
- **Направление «меньше — лучше»** согласовано с остальными метриками: ниже
  коэффициент — меньше денег в час тратится на story point. Метрика поощряет
  ровный, недорогой темп расхода; всплеск затрат за короткое время повышает
  коэффициент.

### Ограничения

- Стоимость — оценка (атрибуция токенов по моделям + официальные тарифы), поэтому
  в UI метрика всегда помечается `≈`.
- Задачи без запусков или без тарифа модели не входят в стоимостные метрики;
  наличие неоценённых токенов помечается флагом (`has_unpriced`, `*`).
- `hours_i` считается по времени запусков; запуск, пересекающий полночь,
  учитывается целиком (то же приближение, что и в атрибуции токенов).
