# AIStat

Статистика использования токенов агентами Multica: сколько токенов, какими
моделями, по каким задачам и проектам, и с какой эффективностью — токенов на
story point, стоимости на story point и «взвешенной» (стоимость в час на 1 SP).
Формулы: [`docs/metrics-efficiency.md`](docs/metrics-efficiency.md).

Данные поступают **только** через аутентифицированный CLI `multica`
(subprocess, `--output json`) — без токенов/ключей в коде и без прямых
HTTP-вызовов к серверу. Локальный поллер пишет один source-файл SQLite;
публичный WSGI хранит изолированный `data/tenants/<user_id>.db` для каждого
пользователя.

Статус: локальный live-дашборд принят QA; дополнительно подготовлен безопасный
публичный WSGI-вариант для Namecheap Shared Hosting с авторизацией и подписанной
синхронизацией данных. Полная инструкция:
[`docs/deployment-namecheap.md`](docs/deployment-namecheap.md).

## Требования

- macOS / Linux, Python ≥ 3.9 (проверено на системном 3.9.6)
- Авторизованный CLI `multica` в `PATH`

## Запуск (одна команда)

```bash
./run.sh
# → дашборд: http://localhost:8787
```

Скрипт сам создаёт venv и ставит зависимости при первом запуске, поднимает
поллер в фоне (лог в `data/poller.log`) и uvicorn с API + дашбордом.
Порт меняется переменной `AISTAT_PORT` (`AISTAT_PORT=9000 ./run.sh`).
Остановка — Ctrl-C (поллер гасится автоматически).

Если настроены `AISTAT_PUBLISH_URL`, `AISTAT_INGEST_SECRET` и
`AISTAT_TENANT_ID`, `run.sh`
дополнительно запускает publisher: не чаще раза в 5 минут он отправляет на
публичный сайт подписанный snapshot SQLite. Multica credential при этом
никогда не покидает локальный компьютер.

## Постоянные локальные развёртки dev / main

Две всегда поднятые локальные развёртки бок о бок (через launchd): ветка `dev`
на `http://127.0.0.1:8788` автоматически подхватывает свежие пуши в `origin/dev`,
ветка `main` на `http://127.0.0.1:8789` обновляется только по запросу. Один
скрипт — [`deploy/local_deploy.sh`](deploy/local_deploy.sh):

```bash
deploy/local_deploy.sh install     # поднять обе развёртки (dev + main)
deploy/local_deploy.sh release     # выпустить dev → origin/main со сборкой и перевыкатить main
deploy/local_deploy.sh status      # состояние, HEAD, health по портам
```

Полная инструкция (обновление dev, релиз в main, логи, удаление, настройка):
[`docs/deployment-local.md`](docs/deployment-local.md).

## Запуск по частям

Поллер (фоновая синхронизация; новый цикл стартует каждые 45 с по умолчанию,
отсчёт от старта до старта — см. «Как работает ингест»):

```bash
.venv/bin/python -m aistat.poller            # бесконечный цикл
.venv/bin/python -m aistat.poller --once     # один цикл и выход (exit 1 при ошибках)
.venv/bin/python -m aistat.poller --once --detail-budget 200   # ускоренный бэкфилл деталей
```

Health-снапшот (счётчики таблиц, последний цикл, ошибки источников):

```bash
.venv/bin/python -m aistat.health
```

HTTP-сервер (API + дашборд, без поллера):

```bash
.venv/bin/uvicorn aistat.server:app --port 8787
```

Установка зависимостей вручную (то же делает `run.sh`):

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

Тесты:

```bash
.venv/bin/pip install -r requirements-dev.txt
.venv/bin/python -m pytest tests/ -q
```

## Дашборд и API (этап 3)

Дашборд (`http://localhost:8787`) показывает: суммарные карточки (токены,
стоимость USD, кредиты, story points, эффективность токенов, эффективность
стоимости и взвешенная эффективность), токены и стоимость по дням
(stacked-графики с группировкой по моделям / агентам / проектам), разбивку по
агентам (doughnut + таблица), проекты (график + таблица со статусами задач, SP и
токены/SP), графики токеновой эффективности по агентам, моделям и времени,
эффективность по моделям (токены/SP, стоимость/SP, взвешенная — дешёвые сверху)
и топ задач по расходу токенов на 1 SP. Графики эффективности при пустом
отфильтрованном результате показывают явное «Нет данных…» вместо пустого
полотна; у каждого есть доступное имя (`role="img"` + `aria-label`) и
раскрываемая таблица «Данные таблицей» с теми же значениями для клавиатуры и
screen reader, а интервалы без данных остаются разрывами (— в таблице).
Фильтры: произвольный период UTC (вплоть до часа), несколько проектов,
агентов и моделей, а также быстрые окна 7/14/30/90 дней/всё время (период
отсчитывается от последней даты в данных). Состояние фильтров сохраняется в
URL и восстанавливается при перезагрузке; параметры из ссылки валидируются
после `/api/meta` до первой загрузки данных — неизвестные ID измерений,
недопустимые `days`/`group`, некорректные (включая календарно невозможные
даты вроде `2026-04-31`, 29 февраля невисокосного года и недопустимое время
вроде `24:00`) и перевёрнутые (`from` ≥ `to`) диапазоны сбрасываются с
видимым пояснением, URL переписывается в нормализованный вид, а кнопка
«Сбросить фильтры» возвращает канонический `/` (FAN-1255, FAN-1269).
Chart.js вендорится локально (`aistat/static/vendor/`), CDN не используется.

**Live-обновления.** `GET /api/events` — SSE-поток: сервер следит за
счётчиком `sync_beats`, который поллер увеличивает после каждой партии
свежих данных — сразу после live-фазы цикла (дневные токены, стоимость,
активность) и после полного цикла. На каждое изменение клиенту уходит
событие `update` (payload: `{beat, cycle}`); после полного цикла
дополнительно шлётся событие `cycle` с записью из `poll_cycles` (для
внешних наблюдателей). Фронтенд по `update` перезагружает данные без
перезагрузки страницы (индикатор «live» в шапке); метка «синхронизация»
показывает время последнего beat'а. Итог: при интервале 45 с открытый
дашборд получает свежие live-данные не реже, чем раз в интервал, даже
если полный цикл (с бэкфиллом деталей) длится дольше.

Эндпоинты API (все отдают JSON):

| Эндпоинт | Что отдаёт |
|---|---|
| `GET /api/meta` | проекты, агенты, модели, диапазон дат (для фильтров) |
| `GET /api/summary?from&to&project&agent&model` | карточки: токены, стоимость, кредиты, SP, токены/SP, стоимость/SP, взвешенная |
| `GET /api/daily?group=model\|agent\|project&from&to&project&agent&model` | серии по дням |
| `GET /api/agents?from&to&project&agent&model` | токены/стоимость/кредиты по агентам |
| `GET /api/projects?from&to&project&agent&model` | разрезы по проектам: токены, стоимость≈, SP, статусы, эффективность |
| `GET /api/efficiency?from&to&project&agent&model&limit` | задачи: токены ÷ SP, худшие сверху |
| `GET /api/model-efficiency?from&to&project&agent&model` | модели: токены/SP, стоимость/SP, взвешенная — дешёвые сверху |
| `GET /api/efficiency-breakdown?from&to&project&agent&model` | графики токенов/SP по агентам, моделям и UTC-времени |
| `GET /api/health`, `GET /health` | health-снапшот (включая unpriced-модели) |
| `GET /api/events` | SSE: `update` после каждой партии данных (live-фаза и полный цикл), `cycle` после полного цикла |

`project`, `agent` и `model` можно передавать повторно (например,
`agent=A1&agent=A2`). `from`/`to` принимают дату (`YYYY-MM-DD`, включительно
для `to`) либо полузакрытый UTC-интервал (`YYYY-MM-DDTHH:MMZ`): диапазон
10:00–11:00 включает ровно этот час.

**Точные и оценочные значения.** Токены по дням/моделям — точные
(совпадают с `multica runtime usage`); токены по задачам/проектам — точные
(совпадают с `multica issue usage`). Разнесение дневных токенов по
*агентам* — точное там, где пара (runtime, model) принадлежит одному агенту,
и оценочное (помечается `≈`/`estimated`) для Codex Dev Sol / QA Codex Sol,
которые делят одну модель и runtime: их доли делятся по длительности
запусков за день (фолбэк — по числу запусков, затем поровну). Разнесение
дневных токенов по *проектам* (фильтр проекта на дневных графиках и
карточках) — всегда оценка по той же схеме. Стоимость проекта — оценка:
точные токены задач тарифицируются по моделям агентов, выполнявших задачу.
Токены, попавшие на модель без тарифа, в стоимость не входят и помечаются.

При неполном дне AIStat распределяет дневную сумму по реальным интервалам
запусков `started_at`–`completed_at` задачи и помечает результат `≈`/
`estimated`. Доли не перенормализуются: если в выбранный час попала четверть
длительности запусков дня, показывается четверть его токенов. Запуски без
обеих дат исключаются из почасовой оценки, а не получают выдуманную
длительность. Это оценка времени работы, а не точные почасовые токены
Multica.

**Эффективность — три метрики** (все: меньше — лучше), полные формулы и
обоснование в [`docs/metrics-efficiency.md`](docs/metrics-efficiency.md):

- **Эффективность токенов** (`tokens_per_sp`) — точные токены ÷ story points;
- **Эффективность стоимости** (`cost_per_sp`) — оценочная стоимость USD ÷ story
  points, с разрезом по моделям (`/api/model-efficiency`), чтобы видеть, какая
  модель делает story point дешевле;
- **Взвешенная эффективность** (`weighted_efficiency`) — стоимость в час на 1 SP
  (USD / час / SP): SP-взвешенное среднее по задачам от `стоимость ÷ активные
  часы ÷ SP`. Формула масштаб-инвариантна, поэтому модели, проекты и периоды
  сравниваются честно.

Все три считаются только по задачам с story points > 0 и загруженной статистикой
токенов; задачи без story points из метрик исключаются (не считаются как 0).
Стоимостные метрики дополнительно требуют запусков задачи (иначе токены нельзя
привязать к модели и тарифицировать) и всегда помечаются `≈` как оценка.
Для фильтра по агенту, модели или времени накопленные токены и story points
задачи получают ту же долю датированных запусков. Стоимость, активные часы и
разрез по моделям строятся из того же отфильтрованного набора пересечений
запусков: модель исключённого агента в разрез не попадает, а часы равны
фактическому времени пересечения с окном (доля фильтра к ним повторно не
применяется). Доля запусков без model metadata остаётся в наборе отдельной
строкой `model = null` и учитывается как неоценённая (`unpriced_tokens`,
`has_unpriced`), а не приписывается известной модели; задача с такой долей
не участвует в weighted efficiency как не полностью тарифицированная. Поэтому task-level
эффективность в таком срезе также помечается как оценочная: `/api/summary`
отдаёт отдельные флаги `sp_estimated` и `efficiency_estimated` (флаг
`estimated` относится только к токенам/стоимости), `/api/efficiency` — флаг
`estimated` на каждую строку, а дашборд показывает `≈` на карточках SP и
«Эффективность токенов» и в оценочных строках таблицы задач. Без фильтров
task-level значения точные и `≈` не получают.

Графики `GET /api/efficiency-breakdown` показывают базовую метрику
`tokens_per_sp` по агентам, моделям и времени. Для каждого запуска его доля
токенов и SP равна доле длительности запуска среди всех валидных запусков этой
задачи: выбранный час или одна модель не раздувается до всей задачи. Явно
заданный UTC-интервал до 48 часов разбит на часы, остальные интервалы — на дни;
результат всегда помечен `≈`. Задачи без датированных запусков остаются в
общей task-level метрике, но в эти графики не включаются, чтобы не выдумывать
время, агента или модель.

**Jira-задачи исключаются из статистики.** Задачи, импортированные из
легаси-архива Jira, несут в metadata Multica маркеры `jira_key` / `jira_url` /
`historical_import` — они не выполнялись в Multica (нет runs и usage). Такие
задачи помечаются флагом `is_jira` при нормализации и не входят в счётчики
задач, суммы story points и эффективность. Нативная задача Multica маркеров не
имеет и учитывается даже если лежит в проекте «Jira Archive» (различие — по
metadata задачи, а не по проекту).

## Подключение своего Мультика

Регистрация/вход — через Google; свой Multica API-токен (PAT) пользователь вводит
вручную в кабинете, и PAT никогда не выступает паролем AIStat. Хост держит токен
только до подтверждённого handoff доверенному локальному worker'у, который хранит
токены зашифрованными. Функция ship'ится **выключенной**
(`AISTAT_MULTICA_CONNECT_ENABLED`), пиннит подключение к официальному хосту, даёт
pending-токену короткий TTL и требует готовый worker перед записью токена.

⚠️ PAT представляет пользователя, видит все его доступные workspaces и обычно не
истекает: заведите отдельный PAT для AIStat (по возможности с ручным сроком) и
обязательно отзовите его в Multica после отключения. Протокол (HMAC pull-канал,
lease/ack, replay-защита), state machine, fail-closed контроли, деплой и residual
risk описаны в [docs/secure-token-handoff.md](docs/secure-token-handoff.md).

После handoff доверенный worker сам собирает статистику по каждому подключению —
логинит официальный CLI под изолированным per-user профилем токеном самого
пользователя, выбирает его workspace, пишет в отдельную tenant-БД и публикует
per-tenant snapshot. Полный per-user pipeline (изоляция профиля, stdin-контракт
токена, пиннинг хоста, выбор workspace, failure isolation, backpressure и
cleanup) описан в
[docs/per-user-collection.md](docs/per-user-collection.md).

## Как работает ингест

Цикл поллера состоит из двух фаз.

**Live-фаза** — данные live-дашборда: `runtime list`, `agent list`,
`project list`, `runtime usage`/`runtime activity` по каждому рантайму и
пересчёт стоимости по тарифам. Сразу после неё поллер коммитит beat
(`sync_beats`), и SSE-клиенты обновляются, не дожидаясь остатка цикла.

**Медленная фаза** — объёмные данные: `issue list` по каждому проекту
(с пагинацией), `agent tasks` по каждому агенту, и — с бюджетом
`AISTAT_DETAIL_BUDGET` за цикл — `issue usage` + `issue runs` по задачам,
у которых изменился `updated_at` (свежие первыми). Бэкфилл деталей —
единственная «эластичная» часть цикла: если тик выходит за дедлайн
(старт цикла + `AISTAT_POLL_INTERVAL_SECONDS`), оставшиеся задачи
откладываются на следующий цикл (но минимум одна задача синхронизируется
за цикл, чтобы бэклог всегда убывал). Отложенные задачи остаются в статусе
stale и не считаются ошибкой источника.

Новый цикл стартует через `AISTAT_POLL_INTERVAL_SECONDS` после старта
предыдущего (а не после его конца), поэтому cadence live-обновлений равна
интервалу, пока не-эластичная часть цикла в него укладывается; при
переполнении следующий цикл стартует сразу. Так активные проекты
обновляются в каждом цикле, а большой легаси-архив (Jira Archive, 1024 задачи)
докачивается постепенно, не раздувая тик.

Все записи — идемпотентные upsert'ы по натуральным ключам: повторные циклы
не создают дублей. Ошибка любого источника логируется, сохраняется в
`sync_state` и видна в health (`status: degraded`, `failing_sources`);
остальные источники цикла продолжают работать, нулевые значения вместо
ошибок не подставляются.

## Схема данных (SQLite)

На публичном хосте каждая статистическая БД имеет ту же схему, но хранится
отдельно: `AISTAT_TENANTS_DIR/<user_id>.db`. В `security.db` таблица `tenants`
фиксирует зарегистрированных владельцев БД, per-tenant replay watermark и
время/SHA последнего принятого snapshot. Путь никогда не строится из email,
URL или произвольной строки запроса — только из числового `users.id`, найденного
в реестре.

Подписанный ingest использует обязательный заголовок `X-AIStat-Tenant`; tenant
ID входит в HMAC `v2`, поэтому подмена только заголовка не может направить
snapshot в чужой файл. Отсутствующая БД зарегистрированного тенанта считается
валидным пустым состоянием.

Одноразовый перенос старой публичной `AISTAT_DB_PATH` в тенант владельца:

```bash
python3 -m aistat.migrate
```

Команда идемпотентна, использует согласованную SQLite-копию, сохраняет rollback
копию `<AISTAT_DB_PATH>.migrated` и печатает `owner_user_id`, который нужно
задать локальному publisher как `AISTAT_TENANT_ID`.

- **Измерения**: `runtimes`, `agents` (модель, runtime), `projects`,
  `issues` (+ `story_points` из metadata с фолбэком на лейбл `SP:N`,
  `estimation_model`).
- **Факты**: `daily_usage` — токены по (runtime, model, date) — 4 вида
  токенов: input / output / cache read / cache write; `issue_usage` —
  суммарные токены и число задач по issue; `runs` — задачи Multica
  (атрибуция issue↔агент↔runtime, статусы, времена); `runtime_activity` —
  почасовой снапшот активности.
- **Служебные**: `sync_state` (здоровье источников), `poll_cycles` (журнал
  циклов), `sync_beats` (однострочный счётчик «данные обновились» для SSE).

Потокенной статистики по отдельным задачам у Multica нет (проверено);
атрибуция токенов агентам делается через (runtime_id, model, date) +
карту агент→(runtime_id, model) — это работа этапа 3. Агенты Codex Dev Sol
и QA Codex Sol делят одну модель и один runtime, поэтому их раздельные
значения будут помечаться как оценочные.

## Тарифы, стоимость и кредиты (этап 2)

Стоимость токенов считается по **официальным тарифам вендоров**. Тарифы лежат в
версионируемом файле [`pricing.json`](pricing.json): для каждой модели — ставки
за 1M токенов раздельно по четырём видам (`input` / `output` / `cache_read` /
`cache_write`), валюта (USD), URL официального источника и дата снятия.

Источники (сняты **2026-07-15**):

| Модель | Вендор | input | output | cache read | cache write (5m) | Источник |
|---|---|---|---|---|---|---|
| `claude-opus-4-8` | Anthropic | $5 | $25 | $0.50 | $6.25 | docs.claude.com/pricing |
| `claude-fable-5` | Anthropic | $10 | $50 | $1.00 | $12.50 | docs.claude.com/pricing |
| `claude-haiku-4-5-20251001` | Anthropic | $1 | $5 | $0.10 | $1.25 | docs.claude.com/pricing |
| `gpt-5.6-sol` | OpenAI | $5 | $30 | $0.50 | $6.25 | developers.openai.com/api/docs/pricing |
| `gpt-5.6-terra` | OpenAI | $2.50 | $15 | $0.25 | $3.125 | developers.openai.com/api/docs/pricing |

Точные URL и даты для каждой ставки — в `pricing.json` и в таблице
`model_pricing` (видна в health, поле `pricing.rates`). Все модели из реальной
БД имеют официальный тариф; список `unpriced`-моделей в health пуст. Если
появится модель без официального тарифа (внутреннее/неизвестное имя), она
**не** оценивается в 0, а помечается `unpriced` (стоимость `NULL`) и выводится
в health (`pricing.unpriced_models_in_usage`).

**Формула стоимости строки usage** (четыре счётчика токенов — независимые
слагаемые; `input_tokens` — это уже некешированный остаток):

```
usd = (input·input_rate + output·output_rate
       + cache_read·cache_read_rate + cache_write·cache_write_rate) / 1e6
```

`cache_write` тарифицируется по ставке кеш-записи с TTL 5 минут (у Anthropic и у
gpt-5.6 это 1.25× input — дефолтный ephemeral-кеш, которым и пользуются агенты);
ставка 1-часового кеша хранится в `pricing.json` (`cache_write_1h`) для справки и
переопределения. `cache_read` — по ставке кеш-хита (0.1× input). У codex-моделей
Multica отдаёт `cache_write_tokens = 0`, поэтому ставка кеш-записи OpenAI на
текущую стоимость не влияет.

Стоимость (`cost_usd`) и кредиты (`cost_credits`) пишутся рядом с usage-строками
в таблицу `daily_usage` и пересчитываются идемпотентно каждый цикл поллера
(источник `pricing` в health). Агрегации по дням/агентам/проектам — этап 3.

**Кредиты.** Официальной публичной модели кредитов у Multica нет, поэтому курс
задаётся допущением: конфигурируемый `credits_per_usd` (env
`AISTAT_CREDITS_PER_USD`, по умолчанию **1.0** → 1 кредит = $1). Кредиты =
стоимость USD × курс.

**Переопределение тарифов без изменения кода.** Правьте `pricing.json`
напрямую, либо укажите `AISTAT_PRICING_OVERRIDES=/path/to/overrides.json` —
второй файл той же структуры (`{"models": {...}}`) переопределяет и дополняет
базовые тарифы (в т.ч. может пометить модель `unpriced` или задать тариф новой).

## Известные ограничения

- Дневная история usage начинается 2026-07-12 (workspace создан 2026-07-11) —
  более старых данных не существует на стороне Multica.
- `runtime activity` отдаёт срез «час → число задач» без даты; хранится как
  заменяемый снапшот.
- Удаление/архивация сущностей на сервере не отслеживается (только upsert).
